  <files>

  <action>
    Implement `deleteConversation(peerPubkey: string): Promise<void>` in ChatAPI per CONTEXT.md locked decision ("Eliminazione conversazione a 3 livelli"):

    **Level 1 -- Local deletion:**
    - Call `messageStore.deleteMessages(conversationId)` to remove all messages for this conversation from IndexedDB.
    - Remove the synthetic dialog from the display bridge (dispatch `dialogs_multiupdate` with the dialog removed).

    **Level 2 -- Gift-wrapped peer notification:**
    - Collect all stored eventIds for this conversation (before deleting from IndexedDB, read them first).
    - Create a gift-wrapped notification message to the peer. Use `wrapNip17Message()` to wrap a kind 14 rumor with:
      - `content: ''` (empty)
      - tags: `['delete-notification']` + `['e', eventId]` for each message eventId being deleted
    - Publish via `relayPool.publishRawEvent()`. The peer's client should listen for this and hide/ignore the referenced messages.
    - This notification is indistinguishable from a regular gift-wrapped message on the relay (per CONTEXT.md: all control events travel as NIP-17 gift-wrap).

    **Level 3 -- NIP-09 relay deletion request:**
    - Create a kind 5 (NIP-09) event referencing the message eventIds via `e` tags.
    - Sign with user's private key and publish to all relays via `relayPool.publishRawEvent()`.
    - This is best-effort -- relays may or may not honor deletion requests.
    - Note: the kind 1059 gift-wrap events are what gets referenced in the NIP-09 deletion, not the inner rumors (relays only see the outer events).

    **Incoming delete notification handling:**
    - In `handleRelayMessage()`: detect incoming rumors with `delete-notification` tag. Extract referenced eventIds from `e` tags. Call `messageStore.deleteMessages(conversationId, eventIds)` to hide those messages locally. Do NOT send a delivery receipt for delete notifications.
