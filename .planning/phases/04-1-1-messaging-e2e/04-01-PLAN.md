Output: Working gift-wrap send/receive for text messages, message store, passing tests.

       - Update `publish()` to use gift-wrap pipeline: instead of sending plaintext to each relay's `storeMessage()`, call `wrapNip17Message()` once, then publish the resulting events via `publishRawEvent()` on each write relay.

       - DB name: `nostra-messages`, version 1, store `messages`
       - Key: auto-increment, indexes: `conversationId` (compound: sorted concatenation of two pubkeys), `timestamp`, `eventId` (unique)
       - Interface `StoredMessage { eventId: string; conversationId: string; senderPubkey: string; content: string; type: 'text' | 'file'; timestamp: number; deliveryState: 'sending' | 'sent' | 'delivered' | 'read'; fileMetadata?: { url: string; sha256: string; mimeType: string; size: number; width?: number; height?: number; keyHex: string; ivHex: string } }`
       - `saveMessage(msg: StoredMessage): Promise<void>` -- upsert by eventId (put if exists)
       - `getMessages(conversationId: string, limit?: number, before?: number): Promise<StoredMessage[]>` -- returns messages sorted by timestamp desc, limit defaults to 50. `before` is a timestamp for pagination.
       - `getLatestTimestamp(conversationId: string): Promise<number>` -- for relay backfill `since` filter
       - `deleteMessages(conversationId: string, eventIds?: string[]): Promise<void>` -- delete specific messages or entire conversation
       - `getConversationId(pubkeyA: string, pubkeyB: string): string` -- deterministic: sort both hex pubkeys alphabetically, join with ':'
       - `getAllConversationIds(): Promise<string[]>` -- returns all distinct conversationId values from the store (needed by backfill to know which conversations to query)
       - Export singleton `getMessageStore()` that lazily opens the DB

    2. **chat-api.ts** -- Integrate gift-wrap pipeline and relay backfill:
       - In `sendMessage()`: instead of `JSON.stringify({...})` + `relayPool.publish()`, call `relayPool.publish(recipientPubkey, content)` which now uses gift-wrap internally (from Task 1). The ChatAPI doesn't need to know about gift-wrap -- it delegates to the pool.
       - Add `sendFileMessage(type, url, sha256, key, iv, mimeType, size, dim?)` method for media (consumed by Plan 02). This creates a kind 15 rumor content string with the file metadata tags.
       - Remove the `sendMedia(type, dataUrl)` method that sent base64 data in event content -- replaced by Blossom upload in Plan 02.
       - In `handleRelayMessage()`: update to handle the new DecryptedMessage format which now includes rumor kind and tags. If rumor kind is 15 (file message), extract file metadata from tags and set ChatMessage.type appropriately.
       - Wire message-store: on send, save to store with status 'sending'. On relay confirm, update to 'sent'. On incoming message, save to store with status 'delivered'.
       - Add `loadHistory(peerPubkey: string, limit?: number, before?: number)` method that reads from message-store (instant local cache).
       - Update `ChatMessageStatus` type to include 'read' state: `'sending' | 'sent' | 'delivered' | 'read' | 'failed'`

       **Relay backfill on init/reconnect (MSG-02):**
       - Add `backfillConversations(): Promise<void>` method. This is called:
         (a) During ChatAPI initialization (after relay pool is connected)
         (b) On relay reconnect event (listen to pool's `connect` or `reconnect` event)
       - Implementation: get all known conversation IDs from `messageStore.getAllConversationIds()`. For each conversation, call `relayPool.getMessages({kinds: [1059], '#p': [myPubkey], since: messageStore.getLatestTimestamp(conversationId)})`. Unwrap and merge results into message-store via `handleRelayMessage()`, deduplicating by eventId.
       - Per CONTEXT.md: "All'apertura app: richiedi ultimi 50 messaggi per ogni chat attiva dal relay pool." Limit the relay query to 50 events per conversation to avoid flooding.
       - Run backfill in background (non-blocking) -- dispatch a rootScope `nostra_backfill_complete` event when done so the display bridge can refresh.

    3. **offline-queue.ts** -- Migrate flush to gift-wrap:
       - The flush path calls `relayPool.publish()` which now internally uses gift-wrap (from Task 1). No changes needed to the flush logic itself.
       - Add exponential backoff retry per CONTEXT.md: base 2s, multiplier 2x, max 5min, max 20 attempts. Currently uses simple retry -- replace with backoff schedule.
       - Add `retryCount` field to QueuedMessage interface.
       - On flush failure, increment retryCount and compute next delay: `Math.min(2000 * 2^retryCount, 300000)`
       - After 20 failures, mark message as 'failed' but keep in queue for manual retry via `retryMessage(id)` method.
